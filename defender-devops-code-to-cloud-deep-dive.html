Securing the DevOps repository → provisioning → cloud pipeline

A single-use-case deep dive into Microsoft Defender for Cloud's DevOps security and code-to-cloud capabilities — how a commit in a GitLab, GitHub or Azure DevOps repository is scanned, how infrastructure-as-code provisioning is hardened, how AI identifies and remediates vulnerabilities, and how a code finding is traced all the way to a running cloud resource holding PII / PCI data.
Use case: Code-to-Cloud DevOps security IaC / provisioning AI remediation Attack paths to PII/PCI
SECTION 1

The use case

Most posture tools can tell you that a storage account is unencrypted. The harder, more valuable question is: which line of code, in which repository, provisioned that storage account — and does it hold regulated data? This deep dive is about closing that loop.

What we are validating

This evaluation focuses on one end-to-end scenario rather than a broad feature sweep: protecting the path that runs from a developer's commit, through infrastructure-as-code (IaC) provisioning, into live cloud resources. The concrete requirements driving the test:

  • Live and production environments. DevOps connectors auto-discover every repository in an organization — including the repos that provision production — so the evaluation reflects real, running estates, not just a sandbox.
  • Containers used for test scenarios. Container images built in the pipeline are scanned and mapped from registry to running workload, so container risk is part of the same code-to-cloud story.
  • Beyond detection — the protection part. Detecting an unencrypted resource is table stakes. This use case exercises the protective controls: blocking a pull request, breaking a build, secret push-protection, and required reviewers — stopping the misconfiguration before it ever provisions.
  • DevOps platform coverage, including GitLab. The estate spans more than one SCM. GitLab integration is explicitly in scope — Section 4 gives the accurate, current picture of what the GitLab connector does and does not do.
  • Data types — the file attack path to PII / PCI. The goal is to follow a finding to where it actually matters: a resource that ends up exposing personally identifiable information or cardholder (PCI) data, surfaced as an attack path.
  • Malicious code & code analysis. Static analysis (SAST), dependency and container scanning, and malware analysis on artifacts — identifying vulnerable or dangerous code in the repo.
  • AI-assisted vulnerability detection & automated remediation. Using the AI capabilities to not only find vulnerabilities in code and IaC but to generate the fix — an auto-created pull request a developer reviews and merges.
The core idea: “shift left” meets “code-to-cloud” Defender for Cloud works in two directions at once. Shift left: it scans code and IaC inside the pipeline so risk is caught before deployment. Code-to-cloud: it correlates a running cloud resource back to the exact repository, file and template that created it. The result is a closed loop — a risky cloud resource can be traced to its source, and a source fix can be proven to remove the cloud risk.

Detection vs. protection — why the distinction matters

CapabilityDetection (visibility)Protection (prevention)

The left column is what every posture tool offers. This use case is built to demonstrate the right column.

SECTION 2

Architecture — how it fits together

Five layers cooperate. Source platforms host the code and IaC; scanners run inside the pipeline (or agentlessly); a DevOps connector ingests the results into Defender for Cloud; Defender for Cloud correlates everything in the cloud security graph; and the provisioned cloud estate is the thing ultimately being protected. Two flows wrap around the stack — code-to-cloud mapping down the right, and the AI-powered remediation feedback loop up the left.

SECTION 3

Process flow — a finding's journey

The architecture above is static structure. This is the dynamic story: what happens, in order, from the moment a developer commits code to the moment Defender for Cloud confirms the provisioned resource is safe — and how the loop feeds back.

Developer action Pipeline scanning Defender for Cloud / AI Cloud provisioning
Figure 2 — Process flow from commit to verified-secure provisioning, with the feedback loop.

Stage by stage

  1. Commit & pull request. A developer pushes a change — application code, an IaC template, a Kubernetes manifest, a pipeline definition — and opens a pull request against a protected branch in a connected repository.
  2. Pipeline security scan. The CI/CD pipeline triggers the scanners. With MSDO this is an explicit pipeline task / GitHub Action; with agentless code scanning it happens connector-side on a daily cadence with no pipeline edits. IaC, SAST, secret, dependency and container analyzers all run.
  3. Findings raised. Each analyzer emits results, normalized into SARIF: IaC misconfigurations, code vulnerabilities, hard-coded secrets, vulnerable open-source dependencies, vulnerable container images.
  4. Surface to the developer. Findings appear in two places at once — inline pull request annotations on the exact offending lines (Azure DevOps & GitHub), and as DevOps security recommendations in Defender for Cloud for the security team.
  5. AI-assisted remediation. For code-scanning alerts, GitHub Copilot Autofix proposes a validated code fix. For IaC misconfigurations, Microsoft Security Copilot can generate a corrective pull request directly from the recommendation.
  6. Fix merged. A developer reviews and merges the fix. Protective gates — secret push protection, required reviewers, branch protection, build-breaking on high severity — stop anything still risky from progressing.
  7. Secure provisioning. The corrected IaC runs and provisions the cloud resources. Because the template was fixed at source, the resource is created in a compliant state — not remediated after the fact.
  8. Code-to-cloud correlation. Defender for Cloud maps the running resource back to the template that created it, evaluates its posture, and — if it is exposed and holds PII / PCI data — raises an attack path that names the originating repository and file.

Stage 8 feeds back into stage 1: the attack path (or a security campaign) creates work items in the repo, so the next commit starts from a cleaner baseline. That loop is the point — posture stops being a quarterly audit and becomes a property of the development process.

SECTION 4

Repository connectors — and the GitLab reality

Everything starts with a connector. Defender for Cloud connects to three source platforms, and the choice of platform materially affects which capabilities you get — this is the section that answers the “GitLab isn't quite there” question with specifics.

How a connector works

A DevOps connector is an Azure resource of type Microsoft.Security/securityConnectors. You create it from Defender for Cloud → Environment settings → Add environment. It authorizes against the SCM, auto-discovers every repository in the selected scope, and ingests findings and posture data. There is no per-repository cost — autodiscovery across an entire organization or group is included.

ConnectorOnboards (scope)Onboarding outline
Azure DevOpsOrganizations → all projects & repositories Add environment → Azure DevOps → name / subscription / resource group / region → Configure access → Authorize → choose all existing or all existing + future organizations → Review & generate. The Container Mapping extension is installed automatically.
GitHubOrganizations → all repositories Add environment → GitHub → install & authorize the GitHub app → select organizations / repositories → Review & generate.
GitLabGroups → all subgroups & projects Add environment → GitLab → Configure access → Authorize → choose all existing groups or all existing + future groups → Review & generate. Requires GitLab Ultimate; covers GitLab SaaS (gitlab.com).
Live & production estates Because a connector auto-discovers all repositories in scope, the repos that build and provision production are in scope from the moment you connect. To keep the PoC controlled you can connect at a narrower scope (a single test group/org) first, then widen to “all existing + future” once you are happy.

GitLab: generally available — but with a real parity gap

The GitLab connector is generally available, not preview. It is genuinely useful: GitLab projects get full finding detection — code vulnerabilities, IaC misconfigurations, exposed secrets and open-source dependency vulnerabilities — and those findings are queryable in cloud security explorer. So “GitLab isn't supported” would be wrong. But “GitLab isn't quite there” is partly fair: several advanced and contextualization features are not available for GitLab today.

GitLab verdict for the evaluation GitLab is GA and fully functional for detecting code, secret, IaC and dependency risk, and those findings are queryable. What GitLab lacks today is the contextualization and protection surface: no posture-configuration recommendations, no in-line PR annotations, and no container / IaC code-to-cloud mapping. If the PoC's headline scenarios are PR annotations and IaC-template attack paths, run them on an Azure DevOps connector (the only platform with IaC code-to-cloud mapping), and treat GitLab as a detection-grade connector. Note also: IaC code-to-cloud mapping is Azure-DevOps-only even versus GitHub.

What the connector raises — DevOps security recommendations

Findings surface as recommendations under Defender for Cloud → Recommendations and the DevOps security blade. Representative names:

  • Finding-resolution (all three platforms): “… repositories should have code scanning findings resolved”, “… secret scanning findings resolved”, “… infrastructure as code scanning findings resolved”, “… dependency vulnerability scanning findings resolved”.
  • Posture configuration (Azure DevOps & GitHub only): “GitHub repositories should have protection policies for default branch enabled”, “… force pushes to default branch disabled”, “Azure DevOps service connections shouldn't grant access to all pipelines”, “GitHub organizations should have base permissions set to no permissions or read”, “… should have more than one administrator”, “… secret scanning push protection enabled”, “Azure DevOps repositories should have GitHub Advanced Security for Azure DevOps enabled”.
A scoring quirk worth knowing DevOps security recommendations do not contribute to Secure Score. Track DevOps progress through the DevOps security blade and governance rules rather than expecting the score needle to move.

Licensing — what is free and what needs Defender CSPM

SECTION 5

Provisioning security — scanning infrastructure as code

“Provisioning” in a modern estate is infrastructure as code. An ARM, Bicep or Terraform file is the blueprint for a cloud resource. Scan the blueprint and you stop the misconfiguration before a single resource exists.

Why scan IaC instead of waiting for the resource

If a Terraform file declares a storage account with public blob access, a posture tool will eventually flag the deployed account — after it is live and possibly already holding data. Scanning the IaC catches the same defect while it is still a few lines in a pull request, where the fix is cheap, reviewable and cannot leak anything. This is the “shift left” half of the loop.

What runs the scan — Microsoft Security DevOps (MSDO)

Microsoft Security DevOps (MSDO) is a command-line application that installs, configures and runs a curated set of open-source static-analysis analyzers, then aggregates everything into a single SARIF report. It ships two ways:

  • Azure DevOps extension — the MicrosoftSecurityDevOps@1 pipeline task.
  • GitHub Action — microsoft/security-devops-action.

There is also agentless code scanning (preview): the connector scans repositories on a daily cadence with no pipeline edits at all — lower friction, but it cannot break a build.

MSDO pipeline task — Azure DevOps

# azure-pipelines.yml
steps:
  - task: MicrosoftSecurityDevOps@1
    displayName: 'Microsoft Security DevOps scan'
    inputs:
      categories: 'IaC,secrets,code,containers'   # scope the scan
      # breakOnSeverity is enforced via branch policy / pipeline gates

MSDO GitHub Action

# .github/workflows/msdevopssec.yml
jobs:
  sample:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Microsoft Security DevOps
        uses: microsoft/security-devops-action@latest
        id: msdo
        with:
          categories: 'IaC,code,containers'
      - name: Upload results to the Security tab
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: ${{ steps.msdo.outputs.sarifFile }}

From detection to protection

The protective controls turn an IaC finding into a hard gate:

  • Break the build — fail the pipeline on findings at or above a severity threshold so a misconfigured template cannot proceed to terraform apply / deployment.
  • Branch policy — require the MSDO scan to pass before a PR can complete, and require minimum two-reviewer approval for code pushes.
  • PR annotations — the finding is pinned to the offending line so the reviewer cannot miss it before approving.
This is the “above and beyond detection” part Detection tells you the storage account will be public. Protection — build-breaking plus a branch policy — means the public storage account is never provisioned at all. Same finding, completely different outcome.

SECTION 6

Code & dependency analysis — finding malicious and vulnerable code

IaC governs the infrastructure; the application code governs what runs on it. This section covers static code analysis (SAST), secret scanning, dependency / open-source scanning and malware analysis of build artifacts.

Two sources of code findings — and how they differ

This is the most commonly misunderstood part of Defender for Cloud DevOps security. Code-vulnerability findings can come from either of two engines, and they are not the same:

🧰

MSDO open-source analyzers

The analyzers bundled in MSDO / agentless scanning. Surface in the DevOps security blade without a GitHub Advanced Security licence. Good breadth, lighter depth.

🔎

GitHub Advanced Security (GHAS / GHAzDO)

CodeQL-powered deep SAST, plus native secret scanning and dependency review. Required for native GitHub code-scanning alerts and for code / secret / dependency findings on Azure DevOps repos. This is the engine behind Copilot Autofix.

The MSDO code & container analyzers

AnalyzerTargetCategory
BanditPython sourceCode / SAST
ESLintJavaScript sourceCode / SAST
BinSkimCompiled binaries (Windows PE, ELF)Binary analysis — insecure compiler / linker settings
TrivyContainer images, filesystems, dependenciesVulnerability & dependency scanning
AntiMalwareBuild artifacts (Windows, via Defender for Endpoint)Malware scan — catches malicious files in the repo / artifacts

The AntiMalware analyzer is the direct answer to “test for malicious code”: it scans artifacts for known malware during the pipeline run. The SAST analyzers (Bandit, ESLint, and CodeQL via GHAS) cover the other half — vulnerable code: injection flaws, unsafe deserialization, weak crypto, path traversal and so on.

Secret scanning — an important currency note

The legacy CredScan secret analyzer was deprecated in September 2023 and removed from MSDO. Repository secret scanning is now delivered through GitHub Advanced Security (GitHub) and GitHub Advanced Security for Azure DevOps (Azure DevOps). The decisive protective control here is secret push protection: a commit that contains a detected secret is rejected at push time — the secret never reaches the remote repository at all.

Don't plan the PoC around CredScan If older guidance mentions CredScan as the MSDO secret tool, it is out of date. Validate secret scanning and push protection through GHAS / GHAzDO instead.

Dependency & open-source component scanning

Modern applications are mostly other people's code. Dependency scanning is delivered by GHAS / GHAzDO (Dependabot and the dependency graph) and by Trivy within MSDO / agentless scanning. It flags vulnerable open-source packages and surfaces the recommendation “… repositories should have dependency vulnerability scanning findings resolved”. Dependabot can additionally open update pull requests automatically.

Where the findings land

  • In the repo — GitHub Security tab code-scanning alerts; Azure DevOps Advanced Security results; inline PR annotations.
  • In Defender for Cloud — the DevOps security blade and recommendations (“… repositories should have code scanning findings resolved”), aggregated across every connected repository so a security team gets one cross-platform view.
SECTION 7

AI-powered detection & automated remediation

Finding a vulnerability is only half the value. The differentiating capability is AI that writes the fix — turning a finding into a reviewed, mergeable pull request without a developer starting from a blank editor.

There are two distinct AI remediation capabilities in this use case. They solve different problems, use different engines, and are surfaced in different places. Keeping them separate avoids a lot of confusion in an evaluation.

A · Microsoft Security Copilot — IaC remediation

This targets infrastructure-as-code misconfigurations. From the recommendation “… repositories should have infrastructure as code scanning findings resolved” in Defender for Cloud:

  1. Open the IaC recommendation and select Reduce risk with Copilot.
  2. Choose Help me remediate this recommendation.
  3. Pick the specific failing security check.
  4. Review the summary Copilot produces and select Submit.
  5. Copilot auto-generates a pull request in the source repository that corrects the IaC template — a developer reviews, approves and merges it.

Generic (non-IaC) recommendations additionally support Summarize with Copilot and Fix with Copilot, which produce a natural-language explanation and remediation guidance, with an optional script to run.

Requirements Security Copilot remediation requires Microsoft Security Copilot to be provisioned with Security Compute Units (SCUs) and appropriate Copilot access in the tenant. Treat it as a capability to enable deliberately for the PoC, not something on by default.

B · GitHub Copilot Autofix — code-scanning alerts

This targets application code vulnerabilities — the CodeQL code-scanning alerts from GitHub Advanced Security. It is part of the GHAS integration with Defender for Cloud (a capability set that was significantly expanded in November 2025).

🤖

Copilot Autofix

For a code-scanning alert, Copilot Autofix and the GitHub Copilot coding agent generate a validated code fix, shown inside the GitHub Security tab for the developer to accept.

📦

Multi-fix merge

Multiple individual fixes can be bundled into a single pull request, so a backlog of alerts is cleared in one reviewable change instead of dozens.

🎯

Security Campaigns

A security admin can launch a GitHub Security Campaign from Defender for Cloud, prioritized by runtime context; assignees use the Copilot coding agent to fix at scale.

🧭

Runtime context in GHAS

Defender for Cloud pushes runtime signals — Internet Exposure, Sensitive Data, Critical Resource, Lateral Movement — into the GHAS dashboards so developers fix the alerts that matter first. Status syncs both ways.

Why this matters for the use case

The requirement was “use the AI features to identify vulnerabilities in code and automate their remediation.” Mapped to the two capabilities:

NeedHow it is met
Identify vulnerabilitiesCodeQL SAST + MSDO analyzers (code), IaC analyzers (provisioning), GHAS secret & dependency scanning — all aggregated in Defender for Cloud.
Prioritize themDefender for Cloud runtime context (exposure, sensitive data, attack paths) is injected into GHAS so the riskiest alerts rise to the top.
Automate remediationCopilot Autofix generates code fixes; Security Copilot generates IaC-fix pull requests; multi-fix PRs and Security Campaigns scale it across the estate.
Keep a human in the loopEvery AI fix is a pull request — reviewed, approved and merged by a developer. AI drafts; people decide.
The mental model AI here is a drafting assistant inside an existing review gate, not an autonomous actor. It compresses the slowest step — writing the patch — while the merge decision, branch policy and required reviewers stay exactly where they were. That is what makes auto-remediation safe to adopt.

SECTION 8

Code-to-cloud — attack paths to PII / PCI data

This is where the use case pays off. A code finding on its own is noise; a code finding that can be proven to expose cardholder data to the internet is an incident. Code-to-cloud is the correlation that turns one into the other.

The mapping — how code is linked to a running resource

  • IaC template mapping. The IaCFileScanner analyzer records which cloud resource each IaC template provisions (Terraform, CloudFormation, ARM, Bicep). Defender for Cloud can then say “this storage account was provisioned by main.tf in repo X.”
  • Container image mapping. The container-mapping extension links an image built in a pipeline to the registry it lands in and the Kubernetes workload that runs it.
  • Both require the MSDO extension and the Defender CSPM plan. IaC template mapping is currently Azure DevOps only; container mapping is Azure DevOps and GitHub.

Querying the link — Cloud Security Explorer

Once mapping data is in the cloud security graph, you can query it. Cloud security explorer ships a built-in query — “Cloud resources provisioned by IaC templates with high severity misconfigurations” — and exposes a “Provisioned by” filter so you can pivot from any resource to its originating template, repository and file.

# Cloud Security Explorer - query intent (built visually in the portal)
Resource type:  Storage account
  + filter:     Internet exposed
  + filter:     Contains sensitive data        (DSPM - PII / PCI)
  + filter:     Provisioned by  ->  IaC template with high-severity misconfiguration
=> returns: exposed, sensitive storage accounts traceable to a specific repo + file

The file attack path to PII / PCI — worked example

This is the “file attack path / where we had PII, for example PCI data” scenario, end to end:

StepWhat happens
CodeA Terraform file in a connected repo declares an Azure Storage account with public network access and anonymous blob access enabled.
ProvisionThe pipeline runs terraform apply; the storage account is created in the cloud, internet-reachable.
Data landsAn application writes customer records — names, card numbers — into a blob container.
DSPM discoversDefender CSPM data-aware security posture (sensitive data discovery) scans the account and classifies it as containing PII / PCI data.
Graph correlatesThe cloud security graph now holds three facts about one node: internet-exposed + contains sensitive data + provisioned by a known IaC template.
Attack pathAttack path analysis raises a path such as “Internet-exposed storage account with sensitive data is publicly accessible” — a single, high-priority, contextual finding.
Trace to sourceBecause of code-to-cloud mapping, the path names the originating repository, the IaC file, and (via CODEOWNERS) the responsible team. A one-click GitHub issue can be created carrying both the runtime and the SDLC context.
Fix at sourceThe fix is made in the Terraform file — ideally via an AI-generated PR — so the resource is never provisioned publicly again. Detection became prevention.
Why the attack path beats a raw finding “Storage account allows public access” might be one of hundreds of recommendations. “An internet-exposed storage account holding PCI data, created by infra/storage.tf in the payments repo” is a single sentence a security lead can act on immediately — and hand to the exact team that owns the file.

Containers in the same picture

The container test workloads follow the identical pattern: an image is scanned for vulnerabilities in the pipeline, container mapping links image → registry → running pod, and an internet-exposed workload running a vulnerable image — especially one with a path to a sensitive data store — becomes its own attack path, traceable back to the Dockerfile and repo.

Prerequisites & boundaries

  • Defender CSPM must be enabled on an Azure / AWS / GCP environment in the same tenant as the DevOps connector — code-to-cloud, attack paths and cloud security explorer are paid-plan features.
  • Sensitive data discovery (DSPM) must be turned on for the PII / PCI classification half of the story.
  • IaC template code-to-cloud mapping is Azure DevOps only today; the GHAS code-to-cloud integration is commercial clouds only.
  • Use only synthetic PII / PCI test data in the PoC — never real cardholder data.

SECTION 10

Considerations & gotchas

Practical points to keep the evaluation accurate and fair — especially in a multi-vendor comparison.

Scoping a multi-vendor comparison

When the same scenario is run against several vendors, compare like for like. The dimensions that actually differentiate products on this use case:

  • SCM breadth. Does the tool connect Azure DevOps, GitHub and GitLab — and at what feature parity? (See the GitLab matrix in Section 4.)
  • Code-to-cloud correlation. Many tools detect IaC misconfigurations. Far fewer map a running resource back to the exact template and repo that created it. That correlation is the test.
  • Attack-path quality. Does the tool produce one contextual path (“exposed + sensitive + provisioned by repo X”) or a flat list of unrelated findings?
  • AI remediation. Does it merely describe a fix, or generate a reviewable pull request? Code and IaC, or just one?
  • Protection, not just detection. Build-breaking, PR blocking, secret push protection — can the tool stop the bad change, or only report it afterwards?

The GitLab question — settled

For an evaluation that includes GitLab: it is GA and reliable for detection (code, secrets, IaC, dependencies) and those findings are queryable. It lacks posture-configuration recommendations, PR annotations, and container / IaC code-to-cloud mapping. So run the headline code-to-cloud and PR-annotation scenarios on Azure DevOps, and judge GitLab on detection coverage — not on features Microsoft has not shipped for it yet. “Isn't quite there” is fair for advanced features, unfair as a blanket statement.

In-pipeline vs. agentless scanning

AspectIn-pipeline (MSDO)Agentless code scanning
SetupAdd a task / Action to each pipelineNone — connector-side, zero pipeline edits
CadenceEvery buildDaily
Can break a build / block a PRYesNo
StatusGenerally availablePreview

For the protection half of this use case you need in-pipeline MSDO — only it can fail a build. Agentless is excellent for broad, low-friction visibility across many repos.

Currency notes — things that have changed

Verify these against your tenant CredScan (the old MSDO secret analyzer) was deprecated in September 2023 — secret scanning is now GHAS / GHAzDO. Agentless code scanning is still in preview. The GHAS ↔ Defender for Cloud integration, including agentic Copilot remediation and Security Campaigns, was significantly expanded in November 2025. DevOps security features move fast — confirm current state in Environment settings before drawing conclusions.

Prerequisites checklist

  • A DevOps connector for each SCM in scope (Azure DevOps / GitHub / GitLab).
  • Defender CSPM enabled on an Azure / AWS / GCP environment in the same tenant — required for PR annotations, code-to-cloud mapping, attack paths and cloud security explorer.
  • GitHub Advanced Security / GHAzDO for deep SAST, secret scanning and Copilot Autofix.
  • Sensitive data discovery (DSPM) enabled for the PII / PCI classification.
  • Microsoft Security Copilot provisioned (SCUs) for AI IaC remediation.
  • The MSDO task / Action added to pipelines that need build-breaking protection.

Production-scope caution

A connector discovers every repo in the selected org / group, production included. That is powerful but means recommendations against production code appear immediately. Start the PoC on a narrow scope, confirm behavior, then widen — and agree internally who triages production findings before they appear.

The MSDO component that maps an IaC template to the cloud resource it provisions.
MSDO
Microsoft Security DevOps — a CLI that runs a curated set of security analyzers in a pipeline and aggregates results to SARIF.
PR annotation
A security finding shown inline on the affected lines of an open pull request (Azure DevOps & GitHub).
Pull request
A proposed, reviewable change to a repository — the unit AI remediation produces and the gate protective controls enforce.
Push protection
A protective control that rejects a commit containing a detected secret at push time, before it reaches the remote.
SARIF
Static Analysis Results Interchange Format — the standard JSON format MSDO uses to normalize analyzer output.
Security Campaign
A coordinated remediation drive, launchable from Defender for Cloud, that assigns prioritized alerts to developers for Copilot-assisted fixing.
Security Copilot
Microsoft Security Copilot — the AI capability that generates IaC remediation pull requests from Defender for Cloud recommendations.
Shift left
Moving security checks earlier — into the repo and pipeline — so defects are caught before deployment.
